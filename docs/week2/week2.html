Why Indexing is Constant O(1) Time

To load an item from a list at index i, the computer does not traverse from the beginning. Instead, it performs a simple arithmetic operation: multiplying the index by the address reference offset and adding it to the starting address.

This multiplication and addition take a fixed number of CPU cycles, regardless of whether you access index 0, index 500, or index 1,000,000. Hence, it executes in constant O(1) time.

Key Takeaway

Python Lists are stored as arrays of pointers. The array contains the addresses of the actual objects. Because the addresses are of a fixed width (8 bytes), calculating the offset is direct and instant.

Amortized Time vs Linear Resize Cost

When a dynamic array is full, appending an element forces the system to perform a resize:

  1. Allocate a new contiguous memory block of larger capacity.
  2. Copy all current elements to the new block (takes O(n) copies).
  3. Reassign reference pointer and add the new element.

By **Doubling** the capacity on resize, copy cost is spread out: resizing becomes rare. In the long run, appending elements costs O(1) on average (amortized).

If we only increase capacity by a **fixed size** (e.g. +1000 slots), the number of resizes grows linearly with `n`. More resizes mean we perform O(n²) total copies, making the average append run much slower.
