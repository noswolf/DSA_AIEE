Why Indexing is Constant O(1) Time

To load an item from a list at index i, the computer does not traverse from the beginning. Instead, it performs a simple arithmetic operation: multiplying the index by the address reference offset and adding it to the starting address.

This multiplication and addition take a fixed number of CPU cycles, regardless of whether you access index 0, index 500, or index 1,000,000. Hence, it executes in constant O(1) time.

Key Takeaway

Python Lists are stored as arrays of pointers. The array contains the addresses of the actual objects. Because the addresses are of a fixed width (8 bytes), calculating the offset is direct and instant.

How Dynamic Arrays Work: The Bus Tour Analogy

Memory is like a parking lot. A standard Array is a contiguous block of reserved parking spots. If you want to grow an array but the adjacent memory spots are occupied, you cannot just expand in-place. You must move to a new, larger location!

The Resizing Overhead (O(n))
When a dynamic array runs out of capacity, it must allocate a brand-new backing array elsewhere in memory, copy all existing elements one by one (taking O(n) operations), and release the old array. This copy step is expensive.
Fixed Resizing: Mini-Bus Tour (+4 seats)
Imagine buying a bus that only adds 4 seats when full. If you have 100 passengers, you will have to buy new buses and copy everyone over 25 times! You waste massive amounts of time copying passengers over and over, resulting in slow quadratic O(n²) total copies.
Doubling Capacity: Double Resizing (Amortized O(1))
Instead of tiny increments, we double the capacity every time we run out of seats (e.g. 1 ➔ 2 ➔ 4 ➔ 8 ➔ 16 ➔ 32...). Resizes become extremely rare. Most elements sit down instantly in pre-reserved empty seats. The copy cost is spread out, making each append run in O(1) on average (Amortized Constant Time).

Core Analogy: The Neighborhood Project

Think of Object-Oriented Programming (OOP) like constructing houses in a residential neighborhood.

Class (The Blueprint)
A Class (like ArrayList) is a house blueprint. It defines what attributes and methods the houses will have, but it is not a physical house itself.
Instance (The House)
An Instance is a real house built from that blueprint (like list1 and list2). Each house has its own individual furniture and rooms (instance variables like size and data) that belong only to that house.
Class Variable (HOA Bylaws)
A Class Variable (like max_size) is like neighborhood association bylaws that apply to all houses. If the HOA updates a rule, every house is instantly affected by it.

The Constructor & self Tracer

Core Analogy: The Factory Setup & ID Badge

When you instantiate an object via obj = ArrayList(), two key things happen:

__init__ (The Setup Assembly Line)
This is a special constructor function that runs automatically the moment a new object is born in memory. It populates the blank object with default parameters (like size = 0 and data).
self (The ID Badge Pointer)
This is a pointer reference. Since you can build multiple separate objects, Python needs to know which specific object is running the code. self translates to "this specific instance right here."

Inheritance & Attribute Resolution (MRO)

Core Analogy: Family Tree Inheritance & Searching

Let's look at how attributes and functions are inherited and resolved:

Inheritance (Parent to Child Traits)
A subclass (like SinglyLinkedList) inherits all traits and functions from its parent base class (like SinglyLinkedListBase). This allows code reuse without rewriting it.
MRO (Method Resolution Order)
Think of MRO like searching for an item in the house. Python first checks the object's bedroom (Instance Dictionary) ➔ If not found, it checks the child's class namespace (SinglyLinkedList) ➔ If still not found, it climbs to check the parent's namespace (SinglyLinkedListBase) until it resolves the attribute or throws an error.
