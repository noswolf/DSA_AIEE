Module 0: Stack & Queue Basics

Stacks and Queues are two Linear Data Structures. Unlike Arrays or Linked Lists where you can access any element via index, these are restricted access structures — they define a strict interface for how data enters and exits the collection.

LIFO — Stack FIFO — Queue Restricted Access Linear Structures

The Stack — Last-In, First-Out (LIFO)

LIFO: The most recently added item is always the first one to be removed.

  • 🔵 Primary Operations: push() to add and pop() to remove.
  • 🔵 Analogy: A stack of cafeteria trays or the "Undo" history in a text editor.
  • 🔵 Access Point: Only the Top element is accessible at any time.
  • 🔵 Real-World Use: Function call stack, browser back button, expression parsing.

The Queue — First-In, First-Out (FIFO)

FIFO: The item that has been in the collection the longest is removed first.

  • 🟢 Primary Operations: enqueue() to add and dequeue() to remove.
  • 🟢 Analogy: A line at a grocery store or a printer's task buffer.
  • 🟢 Access Points: Additions at the Rear; removals from the Front.
  • 🟢 Real-World Use: CPU scheduling, breadth-first search, request queues.

Why Use Restricted Structures?

While Arrays are more flexible, Stacks and Queues provide safety and intent. By using a Stack, you guarantee data is processed in a specific order, which is essential for algorithms like Depth-First Search, backtracking, and expression evaluation. A Queue guarantees fairness — the first task submitted is always the first one processed.

Stack Use Cases
  • Function call stack (recursion)
  • Undo / Redo functionality
  • Balanced parentheses checker
  • Depth-First Search (DFS)
Queue Use Cases
  • CPU / OS task scheduling
  • Breadth-First Search (BFS)
  • Print job spooling
  • Network packet buffering

Module 1: Stack (LIFO)

A Stack is a top-heavy structure with only one entry and exit point: the Top. Think of it like a can of Pringles — chips are added and removed from the same end.

Structure of a Stack

A Stack has only one entry and exit point called the Top. Elements are added and removed only from this point. The Base (bottom) contains the oldest data — it cannot be accessed until everything above it is removed first.

PUSH (Add)

Adds an item to the Top. If the stack is full → Stack Overflow. Time: O(1)

POP (Remove)

Removes the Top item. If the stack is empty → Stack Underflow. Time: O(1)

PEEK / TOP

Returns the value at the Top without removing it. Essential for inspection. Time: O(1)

IS EMPTY

Returns True if the stack has no elements. Always check before Pop. Time: O(1)

Module 2: Stack Implementation

A Stack can be implemented using two underlying data structures: a Python List (Array) or a Linked List. Each has different trade-offs in memory and performance.

1. Array-Based Stack (Python List)

In Python, the built-in list is the most efficient way to implement a stack. Since it is a dynamic array, appending and popping from the end of the list is highly optimized (amortized O(1)).

class Stack: def __init__(self): self._items = [] # Top of stack = end of list def push(self, item): # O(1) amortized — appends to end self._items.append(item) def pop(self): # O(1) — removes last element if not self.is_empty(): return self._items.pop() return None def peek(self): # O(1) — view top without removing if not self.is_empty(): return self._items[-1] return None def is_empty(self): return len(self._items) == 0 def size(self): return len(self._items)

⚠ Important: Always use the end of the list as the "Top". Using list.insert(0, val) or list.pop(0) forces shifting every element — turning an O(1) into O(n).

2. Linked List–Based Stack

Implementing a stack with a Linked List uses the Head pointer as the "Top". New nodes are always inserted at the front, and removal also happens from the front — ensuring truly dynamic growth with no resize overhead.

class Node: def __init__(self, data): self.data = data self.next = None class LinkedStack: def __init__(self): self.top = None self._size = 0 def push(self, data): # Insert at front (O(1)) — new node becomes top new_node = Node(data) new_node.next = self.top self.top = new_node self._size += 1 def pop(self): # Remove from front (O(1)) if self.top is None: return None popped = self.top.data self.top = self.top.next self._size -= 1 return popped def peek(self): return self.top.data if self.top else None def is_empty(self): return self.top is None

Module 3: Stack — Array vs Linked List

Both implementations provide the same logical interface, but their performance characteristics and memory behavior differ in important ways.

Big-O Complexity Comparison

Operation Array-Based (Python List) Linked List-Based

PopO(1)O(1)
Peek / TopO(1)O(1)
Is EmptyO(1)O(1)
Space ComplexityO(n)O(n) +pointer overhead

The Array (*) Asterisk

Array Push is O(1) amortized. When the underlying dynamic array fills up, it triggers a resize (O(n)). For almost all cases this is negligible, but it prevents "Hard Real-Time" performance guarantees.

Space Overhead

Both are O(n) space, but the Linked List has a higher constant factor because it must store a reference/pointer (typically 8 bytes) for every single data element.

When to Choose Which?

  • Python List (Array): Best for most use cases. Simple, fast in practice, cache-friendly memory layout.
  • Linked List: Best when you need guaranteed O(1) with no resize events, or when memory is extremely fragmented.
  • Python's built-in collections.deque is actually a doubly-linked list optimized for O(1) append and pop on both ends.

Module 4: Queue (FIFO)

A Queue is a double-ended linear structure: additions happen at the Rear and removals happen at the Front. Like a real queue — first come, first served.

Structure of a Queue

A Queue has two distinct access points. Elements are added at the Rear (back) and removed from the Front (head). Elements in the middle cannot be accessed directly — they must wait their turn.

ENQUEUE (Add)

Adds an item to the Rear. "Joining the end of the line." Time: O(1)

DEQUEUE (Remove)

Removes the item from the Front. The one waiting longest leaves first. Time: O(1)*

FRONT / PEEK

Returns the value of the Front element without removing it. Identifies "who is next." Time: O(1)

IS EMPTY

Checks if the queue is empty. Must check before Dequeue to prevent errors.

Module 5: Queue Implementation

Array-based queues can be tricky. The challenge is handling empty space left at the front after a removal without causing performance bottlenecks. There are 3 main approaches.

1. Naive Implementation (Shifting Array) ⚠ Slow Dequeue

The simplest approach using list.pop(0). Fundamentally flawed for large data because removing the first element forces every other element to shift by one position.

class NaiveQueue: def __init__(self): self.items = [] def enqueue(self, x): self.items.append(x) # O(1) — append to end def dequeue(self): if not (len(self.items) == 0): return self.items.pop(0) # O(n) — shifts ALL elements!

2. Front-Pointer Implementation (Linear) O(1) Dequeue

Instead of shifting, we simply increment a Front pointer. This makes Dequeue O(1), but creates "Dead Space" at the start of the array that cannot be reused without a reset.

class FrontPtrQueue: def __init__(self): self.items = [] self.front = 0 # tracks the front index def enqueue(self, x): self.items.append(x) def dequeue(self): if self.front < len(self.items): val = self.items[self.front] self.front += 1 # Move pointer, don't shift! return val

3. Circular Queue (Modulo & Resize) Best Array-Based

Wraps the Rear pointer back to index 0 using modulo arithmetic to reuse "Dead Space." When full, the array is copied into a new array with twice the capacity.

class CircularQueue: def __init__(self, capacity=4): self.items = [None] * capacity self.front = 0 self.size = 0 def enqueue(self, val): if self.size == len(self.items): self._resize(len(self.items) * 2) rear = (self.front + self.size) % len(self.items) # wraps around self.items[rear] = val self.size += 1 def dequeue(self): if self.size == 0: return None val = self.items[self.front] self.items[self.front] = None self.front = (self.front + 1) % len(self.items) # shift pointer self.size -= 1 return val def _resize(self, new_cap): new_arr = [None] * new_cap for i in range(self.size): new_arr[i] = self.items[(self.front + i) % len(self.items)] self.items = new_arr self.front = 0

4. Linked List–Based Queue

Maintains two pointers: Front (to remove items) and Rear (to add items). This avoids traversal and ensures every operation is independent of the queue's size.

class Node: def __init__(self, data): self.data = data self.next = None class LinkedQueue: def __init__(self): self.front = self.rear = None def enqueue(self, data): # O(1) — add to rear new_node = Node(data) if self.rear is None: self.front = self.rear = new_node return self.rear.next = new_node self.rear = new_node def dequeue(self): # O(1) — remove from front if self.front is None: return None val = self.front.data self.front = self.front.next if self.front is None: self.rear = None return val
Key Concept: The Rear pointer is the "secret sauce" of the Linked Queue. Without it, we would have to traverse the entire list to find the end — making Enqueue O(n).

The queue's performance depends heavily on how you implement Dequeue. The naive approach has a critical flaw that the other methods solve differently.

Performance Summary

Naive Array: Inefficient because pop(0) forces an element shift. As n grows, performance degrades linearly. Avoid for large queues.

Circular Queue & Linked List: Both provide true O(1) front removal. Circular queue is more cache-friendly; Linked List uses more memory per element.

Module 7: Stack vs Queue

Both are restricted-access linear structures, but their ordering principles make them suitable for completely different problems.

Stack

Last-In, First-Out
  • One access point: Top
  • Add with push()
  • Remove with pop()
  • Process newest first
  • Used in DFS, recursion
  • Undo / expression parsing
VS

Queue

First-In, First-Out
  • Two access points: Front & Rear
  • Add with enqueue()
  • Remove with dequeue()
  • Process oldest first
  • Used in BFS, scheduling
  • Print spooling, networking
